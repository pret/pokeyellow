Arbitrary code execution (ACE, Japanese: 任意コード実行) refers to any method that allows the player to force the game to run custom code written by the player. Usually, this at least requires the game to be executing code from a write-enabled memory region, such as WRAM, SRAM or HRAM on the Game Boy (see Game Boy memory map (http://gameboy.mongenel.com/dmg/asmmemmap.html)), although there exists advanced techniques to bypass this requirement (notably return-oriented programming (https://en.wikipedia.org/wiki/Wikipedia:return-oriented_programming)). If the region executed is manipulable to some extent (e.g., a representation of the player's current party), then an ACE exploit is likely possible.

A typical ACE exploit consists of at least two components, the entry point and the payload. The entry point is where the game first jumps to a user-controlled memory region. Since the game does not normally need to execute code from a write-enabled memory region (aside from special cases such as OAM DMA (/wiki/OAM_DMA)), the entry point is usually a glitch. Well-known examples of ACE entry points include various glitch items (/wiki/Glitch_item) in Generation I, and the Coin Case glitch (/wiki/Coin_Case_glitch) in English Pokémon Gold and Silver (/wiki/Pok%C3%A9mon_Gold_and_Silver).

The payload is the custom code the player wants to ultimately execute. The payload usually resides in a memory region that is easy to manipulate and stable, such as the player's item list, or the names of Pokémon storage boxes. Contents of the payload can range from modifying specific in-game data (such as item IDs and quantities), modifying arbitrary RAM addresses (akin to using a cheating device (/wiki/Cheating_device)), calling in-game functions (e.g., obtaining a Pokémon directly), to completely hijacking the game console and running an unrelated program (e.g., playing Tetris in Pokémon).

Most ACE entry points do not immediately jump to a memory region that is so easy to manipulate. Therefore there is often a third part for an ACE exploit, the bootstrap, which is a way to manipulate data at the immediate jump target to spell out a piece of code to jump a more convenient payload region. The bootstrap can often be the most complicated part of an ACE setup.

## Overview
## 
ACE entry points often involve jump tables (/wiki/Glossary#Jump_table). An invalid index into a jump table will cause the game to interpret out-of-bounds data as the jump target, and that jump target can easily be in the RAM. For example, the effects of different items are usually implemented by different functions, and it is natural to use a jump table, indexed by item IDs, to store the addresses of those functions. If there isn't adequate bounds checking, then glitch items (/wiki/Glitch_item) (with invalid IDs) will jump to invalid entries, and thus potentially ACE. Many other mechanics also use jump tables and are potentially exploitable. For example, in Pokémon Gold and Silver (/wiki/Pok%C3%A9mon_Gold_and_Silver), the text command processor uses a jump table to call different functions for different text commands, and the Coin Case glitch (/wiki/Coin_Case_glitch) executes an invalid text command, giving it ACE potential.

Once the execution pointer reaches the RAM, the difficulty of bootstrapping is primarily determined by the manipulability and length of said RAM region. If a few consecutive bytes are sufficiently manipulable, then one could simply spell out a "jump" instruction to jump somewhere with more space. At the other extreme, some RAM regions are barely manipulable, and one has to be content with "sliding" over them (by executing "safe" instructions that neither help nor hinder) until the execution pointer reaches a region easier to work with. In many in-between cases, it takes some creativity to design a bootstrap. For example, 8F (/wiki/8F) jumps to the start of the party species list, meaning that usable byte values include index numbers of valid Pokémon, but not high values corresponding to long-range jump instructions (unless the player wants to use glitch Pokémon as part of the bootstrap). Therefore established bootstraps first make a short-range jump to a Pokémon's HP value or PP values, then jump to the payload with the 1-byte jp hl instruction.

Games with ACE exploits will usually have "conventional" payload locations, such as the third item in the item pack for Generation I games. Some bootstraps are deliberately designed to jump to an established payload location in order to be compatible with existing codes. Usually, even the "best" payload location is still not fully manipulable by normal means: For example, item lists are limited by available items and the maximum item stack size, and box names are limited by available characters and terminators. A common workaround for this problem is using ACE to modify the payload itself, either directly or indirectly (e.g., using ACE to manipulate an item, then swapping it into the payload). Some advanced setups may include a second stage of bootstrap, allowing the user to write an entire program, and then jump to it.

## Limitations
## 
ACE is often considered the "holy grail" in glitching that can achieve "anything" the player wants to do. However, it is instructive to note the limitations of ACE.

- Complexity: ACE exploits often have complicated setups which can be time-consuming. In many cases, ACE allows the player to do things very efficiently once set up, but the upfront time cost still makes it slower than other glitch methods if the player only wants to do something once. Also, since the bootstrap and the payload involve machine code, it can be difficult to make intuitive sense of them, even for experienced glitchers. Furthermore, even one small mistake can completely invalidate the setup and cause the game to execute unwanted code, which can often crash the game and/or even damage the save file.

- Setup stability: Both the bootstrap and the payload usually correspond to in-game data, so they may be inadvertently changed through gameplay, which will break the ACE setup. For example, 8F (/wiki/8F) bootstraps are tied to the party content, so modifying the party by battling or healing the party (modifying key HP or PP values) or obtaining new Pokémon could break the setup.

- Payload size: Although manipulability problems with the payload could be worked around, the total size of the payload remains a limitation. Especially in early generation games, where the total size of the RAM is small, it may be difficult to find a large enough region that neither affects nor is affected by gameplay in unwanted ways. Even if one is fine with replacing normal Pokémon gameplay entirely, the size of the RAM is still limited compared to the ROM. Plus, a longer payload is generally harder to input and more error-prone. In practice this means that glitchers often need to make use of existing functions in the ROM, sometimes in creative ways, to implement ACE exploits.

- Maintaining control: Unlike cheating devices which can edit the ROM or "lock" a RAM address to a particular value, an ACE exploit that does not completely replace normal gameplay must eventually give control back to unmodified game code in the ROM, and cannot directly affect its behavior. To design an ACE exploit that works continuously, one must find an ACE entry point that can be triggered repeatly, such as map script arbitrary code execution (/wiki/Map_script_arbitrary_code_execution) or OAM DMA hijacking (/wiki/OAM_DMA_hijacking). Even then, the inability to lock a RAM address within a frame means that ACE cannot perfectly emulate a cheating device code affecting the RAM.
