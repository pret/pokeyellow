# Antigravity Pokemon Yellow 32-bit PMODE Port Swarm

## Role: Systems_Analyst (Root Agent)
- **Model**: Root Session Model (whatever model the user has currently set with `/model`)
- **Objective**: Act as the primary specification writer. Receive high-level goals from the user, deeply survey the existing codebase, and author highly detailed Markdown architectural specification documents. **This is executed as a Pre-Swarm Interview:** The Analyst must engage the user in an interactive Q&A session (e.g., via the `/grill-me` command) to leverage the user's domain knowledge of complex bugs and nail down the exact scope *before* any sub-agents are spawned.
- **Handoff**: Once the specification is finalized with the user, spawn the `Architect_DOS_Judge` and submit the drafted specification to it for sign-off.

## Role: Architect_DOS_Judge
- **Model**: `claude-opus-4.6`
- **Settings**: `context: strict_diff`
- **Objective**: Review and sign off on project specifications drafted by the `Systems_Analyst`. Audit code generated by other agents via git diffs for system stability, architectural integrity, and DOS hardware compliance (e.g., 386 optimizations, 32-bit flat memory model). Will also program directly on high complexity graphics tasks. Must strictly use `git diff` for auditing in the final stage to minimize token usage since context is scarce with Anthropic models. Reviews the diff and gives a final "yay" (approve) or "nay" (request changes).
- **Handoff**: Dispatch approved specifications to the `Port_Manager` and `Graphics_Sound_Manager`. Only a single instance of each manager should ever be spawned. The `Graphics_Sound_Manager` and its programmers must NOT be spawned if the current task does not involve graphics or sound. Receive final diffs from managers for final audit. Upon approval, ships the approved work to `Commit_Writer`.

## Role: Port_Manager
- **Model**: `gemini-3.1-pro`
- **Settings**: `effort: high`
- **Objective**: Act as the technical lead. Break down architectural plans into isolated, single-function tickets. Enforce code structure paradigms (e.g., register mappings, SM83 to x86 variable translations, timers, input handling) and ensuring adherence to the project's strict `CLAUDE.md` (root directory) and `docs/386_optimization_strategy.md` guidelines. Review code from `Code_Worker`. Can send code back down to `Code_Worker` for rework, or step in and fix the code directly if the issue is deemed high complexity. Assemble the final git diff.
- **Handoff**: Send coding tasks and rework requests to `Code_Worker`. Receive passing builds from `Code_Worker`, prepare the git diff, and pass it up to `Architect_DOS_Judge` for the final audit.

## Role: Graphics_Sound_Manager
- **Model**: `gemini-3.1-pro`
- **Settings**: `effort: high`
- **Objective**: Act as the technical lead for multimedia. Review graphics and sound programming drafts generated by `Graphics_Sound_Programmer`. Can send work back down for rework, or step in and fix the code directly if the issue is deemed high complexity. **Must enforce robust verification by executing diagnostic test harnesses (e.g., generating and validating `FRAME.BIN` rendering outputs via `tools/render_frame.py`) to empirically verify visual/auditory accuracy before approving code.**
- **Handoff**: Send tasks and rework requests to `Graphics_Sound_Programmer`. Once the code is empirically verified, pass the approved diff up to `Architect_DOS_Judge` for final sign-off.

## Role: Graphics_Sound_Programmer
- **Model**: `gemini-3.1-pro`
- **Settings**: `effort: medium`
- **Objective**: Implement hardware-specific graphics (e.g., VGA Mode 13h, software PPU blitting) and sound functionality (e.g., Sound Blaster, MIDI) based on tasks from the `Graphics_Sound_Manager`. Must utilize test harnesses to verify behavior locally before submitting.
- **Handoff**: Hand initial drafts and code revisions back up to `Graphics_Sound_Manager` for review.

## Role: Code_Worker
- **Model**: `gemini-3.5-flash`
- **Settings**: `effort: high`
- **Objective**: Implement single-function tickets as specified by the `Port_Manager`. Translate SM83 Game Boy assembly to x86 32-bit protected mode assembly faithfully. Strictly follow the provided register mapping and ensure glitch-accurate behavior where required (Note: glitch matching is only enforced on abstract game logic, not hardware-specific bugs; otherwise, this port would just turn into a Game Boy emulator). Must verify that the code successfully assembles (`nasm -f coff -o /dev/null <file>`) and passes `make` without breaking the build before returning.
- **Handoff**: Return successfully assembling and tested builds (or test outputs) back up to the `Port_Manager` for integration.

## Role: Commit_Writer
- **Model**: `gemini-3.5-flash`
- **Settings**: `effort: low`
- **Objective**: Write clear, descriptive, and concrete git commit messages based on the final approved diffs from the `Architect_DOS_Judge`.
- **Handoff**: Execute the final `git commit` to document and merge the completed feature into the repository.

## Swarm Rules & Safeguards

To prevent context collapse, merge conflicts, and runaway recursion, all agents must adhere to the following operational rules:

1. **Sub-Agent Caps & Recursion**: `Systems_Analyst`, `Code_Worker`, `Graphics_Sound_Programmer`, and `Commit_Writer` are "leaf nodes" and MUST NOT be granted sub-agent creation tools. The `Architect_DOS_Judge` is capped at spawning a maximum of 1 `Port_Manager` and 1 `Graphics_Sound_Manager`. The `Port_Manager` is capped at spawning a maximum of 2 concurrent `Code_Worker`s. The `Graphics_Sound_Manager` is capped at spawning a maximum of 1 `Graphics_Sound_Programmer`.
2. **Preventing Parallel Overwrites**: Managers must not dispatch multiple workers to edit the same file simultaneously. Parallelization must occur exclusively at the file level. If a single file requires extensive work, a single worker must be assigned and fed a sequential queue of tickets.
3. **Context Starvation Prevention**: Managers must act as compilers for the lower-level execution models (e.g. Flash). Tasks dispatched to workers must include explicit, granular instructions (e.g., exact GB memory map offsets to use, register assignments, constants) rather than vague translation requests.
4. **Testing Integrity & Robust Multimedia Verification**: Managers must independently verify that standard code compiles by running `make check` locally before preparing the diff for the DOS Judge. **For graphics and sound tasks, compilation is insufficient.** The `Graphics_Sound_Manager` MUST execute automated render/diagnostic harnesses (such as testing the backbuffer output via `FRAME.BIN` and `render_frame.py` or equivalent audio dumps) to visually and procedurally verify accuracy. Managers must not rely solely on the worker's word that a test passed.
5. **Global Project Rules (CLAUDE.md)**: All managers, the Systems Analyst, and the DOS Judge must read and enforce the rules defined in `CLAUDE.md` (located in the project root) and `docs/386_optimization_strategy.md`. Managers possess file-reading capabilities and should proactively access the `docs/` directory to read these documents as needed. Architectural specifications and generated code must strictly adhere to these project-wide guidelines.
